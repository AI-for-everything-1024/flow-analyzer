# ADR 决策工作流

标准化的架构决策记录过程，确保决策的透明度与持久性。

## 1. 触发时机
当遇到以下情况时必须创建 ADR：
*   引入新的核心技术栈（如：换用不同的数据库、消息队列）。
*   重大的模块化拆分或合并。
*   改变系统间的通讯模式或安全策略。
*   定义全局性的编程规范。

## 2. 编写与评审
1.  **起草**: 架构师或资深开发者基于 `adr-template.md` 编写草案。
2.  **公示**: 将草案提交到项目的 `docs/adr/` 目录并提交 PR。
3.  **评审**: 核心团队成员在 PR 中进行技术讨论。
4.  **批准**: 评审通过后，将状态更新为 `已接受 (Accepted)` 并合并 PR。

## 3. 索引管理
在 `docs/adr/README.md` 中维护一份所有 ADR 的索引列表：
| 编号 | 标题 | 状态 | 最后更新 |
| :--- | :--- | :--- | :--- |
| 0001 | 项目初始化架构选择 | Accepted | 2026-03-28 |
| ... | ... | ... | ... |

## 4. 存储规范
ADR 文件必须作为源码的一部分存放在 Git 仓库中，以便与代码版本同步演进。
