# CI/CD 流水线标准与验证

## 1. 静态分析与 Lint (早期反馈)
*   **Linting**: 每次提交都必须运行。`main` 分支不允许存在警告。
*   **静态代码分析**: 检查代码异味、漏洞和复杂度（例如使用 SonarQube, Ruff, ESLint）。
*   **敏感信息扫描**: 确保没有明文密钥被提交（例如使用 gitleaks）。

## 2. 测试金字塔实施
*   **单元测试 (快速且隔离)**:
    *   目标: 业务逻辑、工具函数。
    *   要求: 行覆盖率 > 80%。必须为 I/O 和数据库使用 Mock。
*   **集成测试 (子系统验证)**:
    *   目标: 数据库交互、API 合约、第三方集成。
    *   要求: 使用 Testcontainers（针对 DB/Redis）或 H2。
*   **端到端/烟雾测试 (系统信心)**:
    *   目标: 关键业务路径（例如：用户登录 -> 下单 -> 支付）。
    *   要求: 在发布到生产环境前，先在预发环境运行。

## 3. 质量门禁 (Quality Gates)
如果满足以下条件，流水线必须失败：
- 单元测试覆盖率低于阈值。
- 检测到严重的 SonarQube 问题。
- 依赖项检查发现高风险 CVE 漏洞。
